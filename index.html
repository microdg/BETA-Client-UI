MicroDG

Distributed Data Retrieval & Generation through Microservices

Services

There are four main services used by MicroDG in the this given use case. Each service can be used indepentantly and is accessed through its own interface. Services can also be combined using the "Process Controller Service Listed Below"


Process Controller Service

The Process Controller Service (PCS) has three primary functions:

  1. Tying a number of Microservics together
  2. Building a composite data object
  3. Recording runtime statistics for analysis

Here the user can search for businesses based on location. Each business will then be geo-tagged and added to a new collection. That collection (forming a new composite object) will be persisted to an AWS S3 bucket where it can be viewed and downloaded by the user.

Base Business Details Servce

The Base Buisiness Details Service (BBDS) allows the user to retrieve basic details pertaining to a a list of businesses residing in an in-memory database. These details include the name, phone number and address of these businesses. The user can retrieve these business based on a range of location based seach criteria such as: town, county and region.






Service Gateway »

Geo Tagging Service

The Geo Tagging Service (GTS) allows the user (human or machine) to retireive the the geolocation coordinates (latitide & longitude) for ant given address. This service piggibacks on the existing Google Maps Geo Coding API. The service accepts a human readable address from the user, formats it for use with the Google API and then pasrses out the desired information which is displayed back to the user.

Storage Service (AWS S3)

The Storage Service (SS) persists the composite data object generated by the Process Controller Service (PCS) to an Amazon Web Service (AWS) S3 Bucket. The composite data object, now a JSON file can be downloaded by the user via a hyperlink which is issued to the user upon PCS completion.

What is MicroDG

MicroDG is a distributed suite of Microservices for the rapid processing and merging of data from multiple sources in a logical manner. The resulting composite data-object is then written to an AWS S3 Bucket where it can be downloaded by the user. (Use Case: Sales/Marketing data generation)


MicroDG aims to demonstrate how we can improve the speed and scalability of cloud based software systems by moving away from the classic monolithic architecture pattern to one based on microservices.


The system allows the user to dynamically generate marketing data based on geographical location. This data is retrieved from a number of sources, processed and combined in logical manner, with the resulting composite object being written to an Amazon (AWS) S3 Bucket. The user is then issued a unique hyperlink which they can use to view and download the generated data object.


The motivation for developing MicroDG was to build a cloud based system of microservices for the purpose of dynamically generating and processing data (in this case sales lead data). The system addresses the need for the automated bulk tagging of freely available business data with a variety of additional information such as location data. This type of task would normally be done manually which is time consuming and not financially viable for many businesses.


Each of the MicroDG services is independent and can be accessed via a RESTful API exposed by a HTTP integration layer written in Scala, Spray and Akka. Multiple services can then be combined using an ESB like process controller. All services are designed to be activated via an AngularJS UI providing the user with convenient data retrieval, visualisation and analytics features.
